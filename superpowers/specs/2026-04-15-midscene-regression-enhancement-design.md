# Midscene Regression Enhancement Design

## 背景

当前项目的 `Playwright` 冒烟回归已经覆盖了主流程操作：

- 打开 dumi demo
- 运行节点
- 锁定 / 解锁
- 删除节点

同时，`Midscene` 已经完成了最小可用接入，但目前只做了一条非常浅的视觉断言：

- 页面中能看到标题“自动化回归测试页”

这说明 `Midscene` 已经可以工作，但它对真实业务回归的帮助还不够大。为了让这条 AI 用例更接近真实功能门禁，需要把它提升到“编辑器状态与关键 UI 语义”这一层。

## 目标

在不新增测试页面、不让 Midscene 接管复杂交互的前提下，增强现有 Midscene 用例的回归价值。

本轮需要满足：

1. 继续复用 `e2e/basic-flow.spec.ts`
2. 继续让 `Playwright` 负责确定性操作
3. 让 `Midscene` 负责视觉语义断言
4. 断言内容至少覆盖：
   - 页面存在“添加标题”按钮和编辑器区域
   - 页面至少展示两个步骤节点
   - 第一节点运行并锁定后处于“可解锁”状态

## 非目标

- 不让 Midscene 执行运行、锁定、删除等复杂操作
- 不把整条主流程改写成纯自然语言自动化
- 不新增额外 demo 页面
- 不在本轮覆盖 AI 优化弹窗的完整多步回归

## 方案选择

采用“Playwright 驱动状态，Midscene 断言语义”的混合方案。

原因：

1. `Playwright` 更适合做精确、稳定的点击和等待
2. `Midscene` 更适合在关键状态点上做“人类视角”的语义确认
3. 混合方案能降低 flaky 风险，同时保留 AI 回归价值

## 断言设计

### 断言 1：页面结构

在页面初始加载完成后，由 Midscene 断言：

- 页面上存在“添加标题”按钮
- 页面上存在编辑器主体

### 断言 2：节点可见性

在初始状态下，由 Midscene 断言：

- 页面中至少能看到两个步骤节点
- 节点内容对应当前 demo 的步骤结构

这里不使用过长提示词，避免模型在多条件组合下误判。

### 断言 3：锁定后状态

由 `Playwright` 完成：

- 第一节点运行
- 锁定第一节点

之后由 Midscene 断言：

- 第一节点所在区域已经进入“可解锁”状态
- 页面上能看到“解锁节点”这一操作语义

## 实现原则

### 1. 拆分提示词

每条 `aiAssert` 只承担一个小目标，不使用又长又复杂的组合句。

### 2. 先等待，再断言

所有 `aiAssert` 前必须先由 `Playwright` 保证页面处于稳定状态，避免模型读取到中间态。

### 3. 保留基础 Playwright 断言

当前已有的精确断言不删除，只在此基础上增加 Midscene 的视觉语义检查。

## 文件范围

主要修改：

- `e2e/basic-flow.spec.ts`

其余文件应保持不变，除非为了提升断言稳定性必须补充少量文案或定位辅助。

## 验收标准

完成后至少满足：

1. `pnpm test:e2e` 通过
2. Midscene 用例不再只断言页面标题
3. Midscene 用例至少覆盖页面结构、节点可见性、锁定后解锁状态三类语义
4. 基础 smoke 用例行为不退化
