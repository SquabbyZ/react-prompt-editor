# Large Dataset Manual Testing Design

## 背景

当前大数据示例已经完成了第一阶段目标：

- 在 `docs` 中提供独立的“大数据示例”
- 支持 `200 / 1000 / 2000` 三档预置规模切换
- 展示基础摘要信息
- 保持 `PromptEditor` 的展开、滚动和基础编辑体验

但仅有预置档位还不够支持用户自行手动测试。对于真实使用者来说，他们除了想“看一个固定 demo”，还希望：

1. 自己输入规模参数生成测试数据
2. 对比不同根节点数和层级深度对体验的影响
3. 明确看到当前数据的总节点数、根节点数和层级树摘要

因此需要在现有大数据示例基础上做第二阶段增强：保留预置模板，同时加入轻量手动配置能力。

## 目标

在不把文档 demo 做成复杂性能控制台的前提下，为大数据示例补充“用户手动测试”能力。

本次增强需要满足：

1. 保留现有 `200 / 1000 / 2000` 预置档位，继续支持快速对比
2. 支持用户手动输入参数生成树数据
3. 手动参数仅包含最必要的三项：
   - 总节点数
   - 根节点数
   - 最大层级
4. 摘要区明确展示：
   - 当前模式
   - 总节点数
   - 根节点数
   - 最大层级
   - 层级树摘要
5. 所有新增 UI 和生成内容都支持中英文切换

## 非目标

- 不做完整性能测试面板
- 不加入 FPS、耗时、内存等 benchmark 指标
- 不开放过多高级参数，例如节点内容长度、分支数权重、随机种子
- 不引入运行、AI 优化、拖拽等无关能力
- 不修改 `PromptEditor` 组件本身的 API 或内部实现

## 方案选择

本次采用“预置模板 + 轻量手动参数”的混合方案。

原因：

- 比只保留预置档位更适合用户自行验证不同结构
- 比只做手动表单更适合新用户快速上手
- 比完整性能控制台更轻，仍然符合文档 demo 的边界

## 文件范围

主要更新：

- `docs/components/examples/large-dataset.tsx`

可能保持不变：

- `docs/components/prompt-editor.md`
- `docs/components/prompt-editor.en-US.md`

如果文档说明需要补一行“支持自定义生成”，可在中英文组件文档对应章节增加一句简短补充，但不新增独立章节。

## 交互设计

示例仍然保持“单文件、单编辑器、轻控制区”的结构，但顶部信息区扩展为三个分区。

### 1. 预置档位区

继续保留：

- `200 节点`
- `1000 节点`
- `2000 节点`

点击后直接替换当前编辑器数据。

该区的职责仍然是：

- 快速体验
- 一键切换
- 直观看不同规模差异

### 2. 自定义生成区

新增一个轻量表单，用于手动测试。

表单仅提供三个输入项：

- `总节点数`
- `根节点数`
- `最大层级`

同时提供一个主按钮：

- 中文：`生成数据`
- 英文：`Generate Dataset`

点击后使用当前表单值重新生成树，并替换当前编辑器数据。

该区不提供更多高级参数，避免用户把文档 demo 理解为性能后台。

### 3. 摘要区

摘要区固定展示以下信息：

- 当前模式
- 总节点数
- 根节点数
- 最大层级

并在卡片下方额外展示一条“层级树摘要”文本。

中文示例：

- `20 个根节点 -> 多层分支结构 -> 最大 4 层`

英文示例：

- `20 roots -> multi-level branching -> max depth 4`

该摘要只用于帮助用户理解当前树的组织方式，不展示复杂指标。

## 当前模式定义

为了让摘要区可理解，需要区分当前数据来源。

### 预置模式

当用户点击预置档位按钮时：

- 当前模式显示对应预置名，例如 `200 节点`
- 摘要中的总节点数、根节点数、最大层级来自当前生成结果

### 自定义模式

当用户点击“生成数据”按钮并成功生成后：

- 当前模式显示 `自定义`
- 英文显示 `Custom`
- 摘要中的信息基于用户输入生成后的实际结果

## 数据生成设计

### 生成原则

手动生成和预置生成必须共用同一套树生成逻辑，避免两种模式下结构特征不一致。

必须保证：

- 不是纯平铺根节点
- 节点分布具有层级
- 总节点数尽量贴近目标值
- 节点标题和内容保持可读性
- 节点内容长度足以触发真实高度计算

### 参数含义

#### 总节点数

表示本次目标生成的节点总量。

#### 根节点数

表示顶层节点数量。

#### 最大层级

表示允许展开的最大树深度。这里的“层级”包含根节点所在层。

例如：

- 最大层级为 `1` 时，只生成根节点
- 最大层级为 `4` 时，允许生成到第 4 层

### 生成策略

生成器按以下思路组织：

1. 先生成固定数量的根节点
2. 再按层逐步为上一层节点分配子节点
3. 当接近目标总量时进行裁切
4. 超过最大层级后不再继续向下生成

这样既能保持结构层级，又能较稳定地达到目标总量。

### 层级树摘要生成

摘要文本不绘制真实树图，而是根据生成参数与最终统计结果输出简洁描述。

最少应包含：

- 根节点数量
- 是否为多层结构
- 最大层级

如后续需要更细粒度，也可以在不增加复杂 UI 的前提下补充分支描述，但不作为本次必须项。

## 国际化设计

所有新增内容必须和当前文档语言一致。

包括：

- 自定义区标题
- 输入框标签
- 按钮文案
- 校验错误提示
- 摘要卡片标题
- 层级树摘要文本
- 自动生成的节点标题和正文

同时继续将对应语言包传给 `PromptEditor`：

- 中文页使用 `zhCN`
- 英文页使用 `enUS`

目标是避免出现“顶部是英文、编辑器内部是中文”或反之的混合语言问题。

## 校验与边界

为避免生成无效或过重数据，需要增加轻量校验。

### 必须校验

1. `总节点数 > 0`
2. `根节点数 > 0`
3. `根节点数 <= 总节点数`
4. `最大层级 >= 1`

### 建议边界

- `最大层级` 上限建议为 `6`
- `总节点数` 上限建议为 `3000`

原因：

- 对 docs demo 足够覆盖测试需求
- 避免用户输入极大值导致页面卡顿
- 保持控件语义清晰

### 错误处理

当输入不合法时：

- 不替换当前编辑器数据
- 在表单区域显示就地错误提示
- 错误提示跟随当前语言切换

不需要弹出全局消息，也不需要复杂错误状态管理。

## UI 边界

新增控件应尽量轻量：

- 使用简单输入框和一个按钮
- 不加入可折叠高级参数面板
- 不加入图表
- 不加入调试日志面板

目标是用户可以“一眼理解这个 demo 支持手动测什么”，而不是学习一个新的工具界面。

## 实现拆分

`large-dataset.tsx` 内部新增或调整以下职责：

1. 当前模式状态管理
2. 预置档位切换
3. 手动参数表单状态
4. 表单校验
5. 通用树生成函数
6. 摘要统计和层级树摘要生成
7. 中英文文案选择

仍然保持在单文件内实现，避免为了文档 demo 过度拆分。

## 风险与控制

### 风险 1：顶部控件过多，破坏示例可读性

控制方式：

- 只保留三个输入项和一个按钮
- 不增加额外高级参数
- 继续让 `PromptEditor` 区域作为主体

### 风险 2：手动参数导致生成结构失真

控制方式：

- 复用已有多层树生成思路
- 统一预置和手动模式的生成器
- 用“最大层级”限制树深，而不是开放复杂分支配置

### 风险 3：手动输入极大值导致文档卡顿

控制方式：

- 为总节点数和最大层级设置上限
- 输入非法时直接拦截，不触发生成

### 风险 4：国际化再次出现局部文案不一致

控制方式：

- 所有新增文本统一收敛到同一份中英文文案对象
- `PromptEditor` 的 `locale` 与自定义区文案共用同一语言判断来源

## 验证方式

实现完成后至少验证：

1. `200 / 1000 / 2000` 三档预置仍可正常切换
2. 用户可手动输入 `总节点数 / 根节点数 / 最大层级` 并成功生成数据
3. 非法输入时不会覆盖当前编辑器数据，并能看到清晰错误提示
4. 摘要区能正确显示：
   - 当前模式
   - 总节点数
   - 根节点数
   - 最大层级
   - 层级树摘要
5. 中文和英文文档下，顶部说明、表单、摘要和节点内容都语言一致
6. `docs build` 通过

## 验收标准

- 用户既能一键体验预置档位，也能手动生成测试数据
- 示例能明确展示总节点数、根节点数和层级树摘要
- UI 保持轻量，不演变为复杂性能面板
- 中英文显示一致且无混合语言问题
- 现有大数据示例能力不回退
