# Callback Platforms Demo Design

## 背景

当前文档已经提供了：

- 配置式 AI 优化示例，适合 OpenAI 兼容接口
- 回调模式的文字说明，适合非标准接口和复杂流式场景

但对于真实用户来说，`onOptimizeRequest` 的学习门槛仍然偏高，尤其是在需要适配 Dify、OpenAI、阿里百炼等不同平台时。用户需要一个可以直接在文档站里操作、理解、测试并最终复制到自己项目中的完整示例。

## 目标

新增 `docs/components/examples/callback-platforms.tsx`，以最低上手门槛展示“多平台回调模式”。

该示例需要同时覆盖两种使用路径：

1. Mock 学习路径
2. 实战接入路径

用户应能够先用 Mock 熟悉回调模式，再切换到真实接口测试；测试成功后，可以直接复制“实战版”的接入代码到自己的项目中。

## 非目标

- 不在当前仓库中实现真实后端代理
- 不抽离通用 SDK 或独立适配库
- 不覆盖所有第三方平台私有协议，只内置 OpenAI、Dify、阿里百炼和一个自定义模板

## 交互设计

示例文件采用“单文件双模式”结构：

### 1. 顶部说明区

说明该示例适用于：

- `onOptimizeRequest`
- 非标准响应格式
- 自定义流式处理
- 多平台适配

同时给出推荐接入方式：前端示例对接自己的后端适配层。

### 2. 模式切换区

提供两个模式：

- `快速体验（Mock）`
- `真实接入（实战）`

默认先展示 Mock，以降低首次接触成本。

### 3. 平台选择区

提供以下平台选项：

- `OpenAI`
- `Dify`
- `阿里百炼`
- `自定义`

平台选择会同时影响：

- 实战模式中的请求说明
- 响应解析逻辑说明
- 可复制代码模板

### 4. PromptEditor 演示区

使用一个预置节点作为统一演示内容。

用户在两种模式下都通过同一个 `PromptEditor` 操作 AI 优化，但内部 `onOptimizeRequest` 逻辑根据模式切换：

- Mock 模式：本地模拟结构化回调与流式增量输出
- 实战模式：读取用户填写的接口参数并发起真实请求

### 5. 参数配置区（仅实战模式）

提供最少必填项：

- `API URL`
- `API Key`
- `Model`
- `是否流式`

可选项：

- 平台类型

目标是让用户只需改这几项即可测试自己的后端。

### 6. 可复制代码区

展示一段“实战版 `handleOptimizeRequest` 模板”。

该模板会根据当前平台动态变化，但遵循统一结构：

- 读取 `request.messages`
- 透传 `request.signal`
- 调用用户接口
- 解析对应平台返回
- 统一回调 `callbacks.onResponse`

用户测试成功后，应能直接复制这一段到自己的项目。

## 实现设计

### 文件

新增：

- `docs/components/examples/callback-platforms.tsx`

更新：

- `docs/components/prompt-editor.md`
- `docs/components/prompt-editor.en-US.md`

将该示例挂到“高级模式（回调式）”附近，作为回调模式的主要示例。

### 组件职责

`callback-platforms.tsx` 内部包含以下逻辑分区：

1. 示例状态管理
2. 模式切换
3. 平台切换
4. 实战参数输入
5. Mock 请求实现
6. 实战请求实现
7. 多平台响应解析
8. 代码模板生成

不再继续拆分额外文件，以减少用户理解和复制成本。

### Mock 模式逻辑

Mock 模式必须做到：

- 不依赖外部接口
- 明确演示多次 `callbacks.onResponse`
- 演示 `done: true`
- 演示 `thinkingProcess`

目标是让用户理解组件期望的回调协议，而不是理解某个平台的私有格式。

### 实战模式逻辑

实战模式必须做到：

- 允许用户填写自己的接口地址
- 允许用户切换平台类型
- 支持流式和非流式两类响应
- 正确处理 `AbortError`
- 在字段解析失败时给出明确提示

### 平台适配策略

统一把不同平台的响应映射成组件内部协议：

- `optimizedContent`
- `thinkingProcess`
- `done`

内置平台解析优先级：

#### OpenAI

- 优先读取 `choices[0].delta.content`
- 非流式时读取 `choices[0].message.content`

#### Dify

- 流式时优先读取 `event === "message"` 对应的 `answer`
- 结束事件用于标记完成

#### 阿里百炼

- 优先读取文本增量字段
- 使用结束标记或完整响应判断 `done`

#### 自定义

- 提供最小兜底解析模板
- 优先尝试 `content`、`text`、`output`、`answer`

## 文档内容要求

在文档中，这个示例需要传达三件事：

1. 为什么要用回调模式
2. 如何先用 Mock 理解协议
3. 如何把实战模板复制到自己的项目

文案必须尽量避免空泛概念，优先面向“复制即可改”的用户心智。

## 错误处理

必须覆盖以下错误路径：

- 未填写 API URL
- 请求失败
- 响应为空
- 平台字段不匹配
- 用户取消请求

其中“用户取消请求”不应被当作错误提示。

## 验证方式

实现完成后至少验证：

1. Mock 模式可正常展示流式回调
2. 模式切换不会破坏编辑器使用
3. 平台切换能影响可复制代码模板
4. 实战模式缺少必要参数时有清晰提示
5. 文档页可正常渲染示例，无类型或编译错误

## 验收标准

- 用户无需外部接口即可通过 Mock 理解回调模式
- 用户可在同一示例里切换到真实接口测试
- 用户可直接复制实战模板到自己的项目
- 示例代码与现有文档结构一致，风格统一
- 中英文文档都能指向该示例
