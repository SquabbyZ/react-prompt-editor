# RPEditor 文档站官网化改造设计

## 背景

当前站点基于 dumi 默认主题构建，已经具备完整的组件文档、示例页和国际化说明页，但整体视觉与信息组织仍偏向默认文档模板：

- 首页缺少明确的产品价值表达，无法充分体现组件库的差异化能力
- 组件页、示例页、说明页之间缺少统一的标题区、内容节奏和视觉规范
- 多数页面仍是“内容堆叠式文档”，品牌感、专业感和站点一致性不足
- 首页文案与项目当前真实功能存在偏差，部分信息已经过时

本次改造目标是将站点升级为“首页像产品官网，正文像高质量技术文档”的风格。

## 目标

- 统一首页、组件页、示例页、说明页的整体视觉语言
- 强化首页产品价值表达，突出 Prompt 工程化和 AI 优化能力
- 保持技术文档的高效检索与阅读体验，不牺牲 API、示例、说明内容的可用性
- 基于 dumi 本地主题能力渐进式定制，避免重写整套默认主题
- 为后续新增页面提供可复用的视觉和信息结构规范

## 非目标

- 不在本阶段改动组件库运行逻辑、示例代码业务逻辑或导出 API
- 不在本阶段重写整套 dumi `GlobalLayout`
- 不做品牌资产的大规模重设计，例如 logo 重绘、插画体系、营销图素材体系
- 不在本阶段引入新的内容管理系统或替换 dumi

## 设计原则

### 视觉原则

- 气质采用“克制专业”
- 白底、留白、轻层级为主，品牌渐变只用于首页首屏和少数强调区
- 优先通过排版、分组、容器和节奏建立高级感，而不是依赖重装饰

### 信息原则

- 首页讲价值，子页面讲使用
- 组件页、示例页、说明页保持高信息密度和可扫描性
- 标题、导读、示例、API、注意事项使用统一的信息层次

### 技术原则

- 内容源继续以 `docs/*.md` 为主
- 视觉包装与布局能力尽量收敛到 `.dumi/theme` 下
- 优先覆盖 `slots`，必要时才进入 `layouts`
- 保持中英文页面结构一致，避免双语风格漂移

## 站点定位

### 首页

首页承担品牌入口与产品说明职责，需要回答以下问题：

- 这是什么组件库
- 它解决什么问题
- 它与普通 Prompt 输入框有何区别
- 从哪里开始试用和接入

首页应具备更强的官网表达，包括：

- 明确的一句话定位
- 核心卖点卡片
- 使用场景摘要
- 快速体验和安装入口
- 文档与仓库的清晰跳转

### 子页面

子页面承担技术文档职责，核心要求是“读得快、找得到、结构统一”。

子页面需要具备以下体验：

- 标题区统一
- 每个页面有导读或摘要
- 示例区有一致的容器样式与说明文案
- API 区有更清晰的表格与章节层次
- 说明页有统一的提示块、注意事项和段落节奏

## 信息架构

### 首页结构

首页采用如下结构：

1. Hero
2. Core Features
3. Use Cases
4. Quick Demo
5. Installation
6. Documentation Links

其中：

- Hero 负责定位、价值主张、CTA
- Core Features 负责展示差异化能力
- Use Cases 负责建立真实使用语境
- Quick Demo 和 Installation 负责转化

### 组件页结构

组件页采用如下结构：

1. 页面标题与一句话说明
2. 样式引入或快速接入提示
3. 快速上手
4. 典型示例分节
5. API
6. 数据结构与高级说明

要求：

- 例子前有清晰标题和使用场景说明
- API 前保留清晰的章节边界
- 表格、代码块、提示信息风格一致

### 示例页结构

示例页采用如下结构：

1. 示例标题
2. 场景说明
3. Demo 展示
4. 关键点说明

目标是让示例页不仅“能跑”，还“易理解”。

### 说明页结构

说明页采用如下结构：

1. 标题
2. 摘要导读
3. 操作步骤
4. 注意事项
5. 相关链接

目标是减少纯文本堆叠，提升可扫描性。

## 主题实现方案

### 第一层：站点级配置

继续使用 [.dumirc.ts](file:///Users/yuanyuan/Desktop/react-prompt-editor/.dumirc.ts) 维护基础站点品牌配置：

- `themeConfig.name`
- `themeConfig.logo`
- `themeConfig.socialLinks`
- `themeConfig.footer`

### 第二层：本地主题覆盖

在 `.dumi/theme` 中进行渐进式覆盖，建议优先实现以下插槽：

- `slots/Hero.tsx`
- `slots/Features.tsx`
- `slots/Footer.tsx`
- `slots/Content.tsx` 或等效正文包装插槽
- `slots/Header.tsx`、`slots/Logo.tsx`、`slots/Navbar.tsx` 视最终需求决定

覆盖策略：

- 首页重点依赖 `Hero` 与 `Features`
- 全站统一主要依赖正文包装和页脚、导航细节
- 保持 markdown 为内容源，不把页面内容硬编码进 TSX

### 第三层：必要时扩展 Layout

只有在 slot 级覆盖无法满足时，才考虑定制 `layouts/DocLayout.tsx`。

触发条件：

- 需要重排首页与文档页骨架
- 需要统一插入额外的全局导读区或页面头部逻辑
- 需要对默认布局中的区块顺序做结构性调整

本次设计不把 `DocLayout` 重写作为默认前提。

## 文案策略

### 首页文案

首页文案从“功能列表”升级为“产品官网表达”：

- 一句话定位要突出“树形 Prompt 编辑器”
- 卖点表达要覆盖节点运行、依赖编排、AI 对话式优化、流式响应、多平台兼容、高性能渲染
- 安装说明要与真实接入方式一致
- GitHub 链接替换占位地址
- 中英文首页要保持相同的信息结构，不做机械直译

### 子页面文案

子页面文案策略如下：

- 首段说明强调组件用途与适用场景
- 示例说明更偏“为什么用”而不只是“怎么写”
- API 文案更偏清晰和精确，避免营销化
- 注意事项、限制条件和扩展方式要独立成块

## 页面视觉规范

### 首页

- 更强的标题层级和留白
- 轻品牌渐变和柔和背景装饰
- 卡片化卖点区
- CTA 明确且数量克制
- Demo 区与安装区保持专业、简洁

### 子页面

- 标题区统一字号、间距、摘要样式
- 正文容器宽度统一
- 示例区使用统一卡片/边框/标题说明结构
- API 表格、代码块、提示块、分割线采用统一视觉规则
- 页面之间保持一致的 section spacing

## 内容迁移策略

- 首页中原有普通 markdown 的卖点列表，需要改造为更结构化的 `hero/features + section content`
- 组件页在不破坏示例代码引用的前提下，补充导读与结构化标题
- 说明页增加摘要块和注意事项块
- 中英文页面同步调整结构，保证双语一致

## 风险与规避

### 风险

- 局部 slot 覆盖后与 dumi 默认布局的衔接不一致
- 首页和子页面视觉语言失衡，导致“首页像官网，内页像默认模板”
- 结构调整过度，影响现有链接锚点与文档阅读效率
- 中英文内容结构不同步，导致维护成本上升

### 规避措施

- 先确定统一的 section、容器、标题和卡片规范
- 优先 slot 级覆盖，减少对 layout 的侵入
- 控制首页装饰强度，保证与子页面风格同源
- 每类页面至少挑一页作为模板页面先落地验证

## 验收标准

- 首页具备明确产品定位、卖点区、场景区、Demo、安装和链接入口
- 组件页、示例页、说明页都能明显感受到统一视觉体系
- 中英文首页和主要文档页结构一致
- 站点整体风格符合“产品官网感 + 克制专业”
- 文档阅读效率不低于改造前
- dumi 原有示例、表格、锚点、语言切换正常工作

## 测试与验证

### 手动验证

- 检查首页中英文切换后的结构与文案完整性
- 检查组件页示例是否正常渲染
- 检查示例页和说明页的标题、间距、卡片、提示块是否统一
- 检查页脚、导航、按钮、链接视觉是否一致
- 检查暗色模式和主题模式相关页面是否仍可正常展示

### 工程验证

- 运行 dumi 本地开发环境查看改造结果
- 检查近期编辑文件的诊断信息
- 必要时补充针对主题文件的类型和渲染校验

## 实施顺序建议

1. 统一首页文案与信息结构
2. 搭建 `.dumi/theme` 本地主题骨架
3. 落地 `Hero / Features / Footer`
4. 统一正文与标题区样式
5. 统一组件页与说明页结构
6. 统一示例页展示与说明结构
7. 全站联调与细节打磨

当前无需新增开放问题。本次设计已经确定以下决策：

- 视觉方向采用产品官网感
